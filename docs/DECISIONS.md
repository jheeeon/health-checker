# Decision Log

이 문서는 `health-checker`의 핵심 의사결정을 기록하는 로그다.

목적:

- 구현 전에 사용자 확인이 필요한 질문을 추적한다.
- 어떤 결정이 언제, 왜 내려졌는지 남긴다.
- 다음 세션의 Codex가 이미 끝난 논의를 다시 반복하지 않게 한다.

## 사용 규칙

- 핵심 기능 관련 항목은 `결정 필요` 상태로 먼저 등록한다.
- 사용자 확인이 끝나기 전에는 `확정`으로 바꾸지 않는다.
- 결정이 완료되면 날짜와 결정 이유를 함께 남긴다.
- 구현은 확정된 결정만 기준으로 진행한다.

상태 정의:

- `결정 필요`: 아직 사용자 확인 전
- `확정`: 사용자와 합의 완료
- `보류`: 나중 단계로 미룸

## 핵심 의사결정 목록

### D-001 배포 플랫폼

- 상태: `확정`
- 질문: 정적 웹페이지를 어디에 배포할 것인가
- 사용자 결정: GitHub Pages를 사용한다.
- 결정일: 2026-05-17
- 이유: Android 개발자에게 GitHub 기반 배포 흐름이 이해하기 쉽고, 한 장짜리 정적 페이지에 충분하다.
- 메모: Cloudflare Pages도 가능하지만 첫 버전에서는 GitHub Pages를 기준으로 문서화한다.

### D-002 동기화 저장소

- 상태: `확정`
- 질문: 기기 간 체크 상태 동기화를 어떻게 처리할 것인가
- 사용자 결정: Supabase를 사용한다.
- 결정일: 2026-05-17
- 이유: GitHub Pages는 정적 호스팅만 제공하므로 원격 저장소가 필요하다. Supabase는 간단한 테이블 저장과 이후 확장에 적합하다.
- 메모: Firebase도 가능하지만 이번 문서에서는 `GitHub Pages + Supabase` 조합을 기준으로 한다.

### D-003 보안 수준

- 상태: `확정`
- 질문: 첫 버전에서 로그인과 접근 제어를 구현할 것인가
- 사용자 결정: 우선 보안은 신경쓰지 않고 퍼블릭으로 생각한다.
- 결정일: 2026-05-17
- 이유: 개인용 실험 단계에서는 구현 속도와 단순성이 더 중요하다.
- 메모: URL을 아는 사람 또는 Supabase 설정을 아는 사람은 데이터를 수정할 수 있다. 민감 정보 저장 전에는 반드시 재검토한다.

### D-004 앱 형태

- 상태: `확정`
- 질문: 여러 화면을 가진 웹앱으로 만들 것인가, 한 장짜리 페이지로 만들 것인가
- 사용자 결정: 한 장짜리 페이지로 만든다.
- 결정일: 2026-05-17
- 이유: 매일 열람하고 체크하는 용도이며, 라우팅이 필요한 복잡한 기능이 없다.

### D-005 저장 방식

- 상태: `확정`
- 질문: 체크 상태를 브라우저 localStorage에 저장할 것인가, 원격 DB에 저장할 것인가
- 사용자 결정: 기기 간 동기화가 필요하므로 Supabase에 저장한다.
- 결정일: 2026-05-17
- 이유: localStorage는 같은 기기/브라우저 안에서만 유지되어 요구사항을 만족하지 못한다.

### D-006 프로그램 리셋

- 상태: `보류`
- 질문: 7일 프로그램을 다시 시작하는 기능을 첫 버전에 넣을 것인가
- 현재 권장안: 첫 버전에서는 제외한다.
- 이유: 리셋은 데이터 삭제 또는 히스토리 설계가 필요해진다. 초기에는 7일 체크와 동기화 검증을 우선한다.

### D-007 장기 통계

- 상태: `보류`
- 질문: 주간/월간 통계와 그래프를 넣을 것인가
- 현재 권장안: 첫 버전에서는 제외한다.
- 이유: 현재 목표는 회복 루틴의 실행 가능성 확인이며, 장기 분석은 후속 기능에 가깝다.

### D-008 체크리스트 데이터 모델

- 상태: `확정`
- 질문: 체크리스트 항목 정의와 체크 상태를 어떻게 저장할 것인가
- 사용자 결정: 체크리스트 정의는 `program_days`, `program_items`로 DB화하고, 체크 상태는 `daily_item_checks`에 저장한다.
- 결정일: 2026-05-17
- 이유: 체크리스트 문구, 순서, 카테고리 변경이 자주 발생할 수 있으므로 프론트엔드 하드코딩과 JSON key 기반 저장은 확장성이 낮다.
- 메모:
  - `program_items.id`는 `bigint identity` 자동 증가 값으로 사용한다.
  - `item_order`는 화면 노출 순서로만 사용한다.
  - `category`는 의미 분류 컬럼으로 유지한다.

### D-009 checked_items 마이그레이션

- 상태: `확정`
- 질문: 기존 `daily_logs.checked_items` 값을 새 `daily_item_checks`로 마이그레이션할 것인가
- 사용자 결정: 마이그레이션은 생략한다.
- 결정일: 2026-05-17
- 이유: 기존 체크 데이터는 유의미한 운영 데이터가 아니며, 새 구조 도입 후 `daily_item_checks`가 정식 체크 저장소가 된다.
- 메모: `daily_logs.checked_items` 컬럼은 더 이상 사용하지 않으며 삭제 대상으로 본다.

### D-010 모바일 새로고침

- 상태: `확정`
- 질문: 모바일에서 최신 동기화 상태를 어떻게 다시 불러올 것인가
- 사용자 결정: 페이지 최상단 pull-to-refresh를 지원한다.
- 결정일: 2026-05-17
- 이유: 모바일 사용자가 별도 버튼 없이 익숙한 제스처로 Supabase 최신 상태를 불러올 수 있다.
- 메모: 커스텀 녹색 인디케이터는 제거하고 브라우저 기본 회색 인디케이터와 저장 상태 텍스트만 사용한다.
